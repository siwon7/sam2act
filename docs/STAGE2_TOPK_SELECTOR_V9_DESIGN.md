# Stage2 Top-K Selector V9 Design

Date: 2026-05-04

## 1. Question

The proposed V9 idea is:

```text
Stage1 keeps top-K candidate peaks.
Stage2/memory selects the candidate that matches the current episode state.
```

This document checks whether that is a reasonable design based on papers,
public repositories, and the current SAM2Act code path.

## 2. Literature and Repository Evidence

### 2.1 Behavior Transformers: candidate mode + correction

Behavior Transformers explicitly target multimodal behavior cloning by
discretizing actions into modes and predicting a continuous correction. This is
the closest behavior-cloning analogy to our Stage1/Stage2 split:

```text
mode/bin selection -> continuous correction
Stage1 peak       -> Stage2 local action refinement
```

Design implication:

- Stage1 should not be forced to output a single final answer under ambiguity.
- Stage1 can output candidate modes.
- Stage2 should refine the selected mode.

Sources:

- Paper: https://arxiv.org/abs/2206.11251
- Repo: https://github.com/notmahi/bet

### 2.2 C2F Q-attention: discrete attention then local zoom

C2F Q-attention uses a discrete attention map to decide where to zoom, then
refines at a finer scale. SAM2Act already uses a similar idea:

```text
Stage1 coarse heatmap -> crop/zoom -> Stage2 local prediction
```

Design implication:

- The two-stage crop formulation itself is well motivated.
- The weak point is using only one coarse location when the coarse map is
  ambiguous.

Sources:

- Paper: https://arxiv.org/abs/2106.12534
- Repo containing ARM/C2F-ARM: https://github.com/stepjam/ARM

### 2.3 Q-attention with Tree Expansion: top-K for coarse ambiguity

Tree Expansion is especially relevant. It states that coarse Q-attention can
suffer from coarse ambiguity and addresses it by expanding a tree over top-K
voxels, accumulating value estimates across branches.

This maps directly to our problem:

```text
QTE:    coarse voxel ambiguity -> expand top-K voxels -> evaluate branches
SAM2Act: Stage1 peak ambiguity -> keep top-K peaks    -> evaluate/select branches
```

Design implication:

- Top1-only crop is the wrong bottleneck when the coarse prediction is
  ambiguous.
- Top-K expansion is not an ad hoc trick; it is a known fix for coarse
  ambiguity in RLBench-style manipulation.

Sources:

- Paper: https://arxiv.org/abs/2204.12471
- Repo: https://github.com/stepjam/ARM

### 2.4 Learned Path Ranking: rank candidate continuations

Learned Path Ranking ranks candidate goal-reaching paths generated by different
modules. Our candidate set is not paths yet, but each Stage1 peak is a candidate
continuation because it defines the Stage2 crop and local action branch.

Design implication:

- A learned selector/ranker over candidate branches is reasonable.
- Ranking should be trained directly, not inferred only from heatmap CE.

Sources:

- Paper: https://arxiv.org/abs/2204.01571
- Repo: https://github.com/stepjam/ARM

### 2.5 PerAct and RVT: discretized action detection is a strong base

PerAct formulates manipulation as detecting the next best voxel action. RVT
keeps the discretized/multi-view action prediction style and improves
efficiency with multi-view transformers.

Design implication:

- We should not discard the current heatmap/action-detection formulation.
- The targeted change is top-K candidate preservation and selection, not a full
  policy rewrite.

Sources:

- PerAct: https://arxiv.org/abs/2209.05451, https://github.com/peract/peract
- RVT: https://arxiv.org/abs/2306.14896

### 2.6 Transporter / CLIPort: spatial affordance maps are useful but top1 is limited

Transporter-style methods learn spatial attention/transport maps for
manipulation, and CLIPort combines semantic and spatial pathways. These support
using heatmaps/affordance maps as action proposals.

Design implication:

- Heatmaps are a good proposal representation.
- For memory-dependent ambiguity, heatmaps need an explicit selection stage.

Sources:

- Transporter repo: https://github.com/google-research/ravens
- CLIPort repo: https://github.com/cliport/cliport

### 2.7 IBC and Diffusion Policy: stronger but heavier alternatives

Implicit BC and Diffusion Policy directly model multimodal action
distributions. They are strong evidence that multimodal imitation is a real
distribution-modeling problem.

Design implication:

- These are valid future directions.
- They are larger changes than needed for the immediate SAM2Act pipeline.
- V9 should keep the current architecture and add explicit top-K selection.

Sources:

- IBC paper/repo: https://arxiv.org/abs/2109.00137,
  https://github.com/google-research/ibc
- Diffusion Policy paper/repo: https://arxiv.org/abs/2303.04137,
  https://github.com/real-stanford/diffusion_policy

## 3. Current SAM2Act Bottleneck

The current stage-two path is top1-only.

During inference, `mvt_sam2.py` gets the Stage1 waypoint with:

```text
wpt_local = self.get_wpt(out, mvt1_or_mvt2=True, ...)
pc, rev_trans = trans_pc(pc, loc=wpt_local, sca=st_sca)
out_mvt2 = self.mvt2(...)
```

`get_wpt()` eventually calls `get_max_3d_frm_hm_cube()`, which averages
multi-view heatmap scores over rendered 3D points and returns the single
maximum point.

So if Stage1 top1 is wrong, Stage2 gets a wrong crop. The rest of the
pipeline may be healthy but has already been boxed into the wrong local
workspace.

The V9 intervention point is therefore precise:

```text
replace get_max_3d_frm_hm_cube()
with get_topk_3d_frm_hm_cube() + learned candidate selection
```

## 4. V9 Design Goal

V9 should answer three separate questions:

```text
1. Coverage: does Stage1 topK contain GT/valid?
2. Selection: if GT is in topK, can memory/Stage2 select it?
3. Refinement: after selection, can Stage2 produce the correct local action?
```

This is better than only reporting final success, because each failure becomes
actionable.

## 5. Proposed Architecture

### 5.1 Candidate extraction

Add a renderer helper:

```python
get_topk_3d_frm_hm_cube(hm, k=3, nms_radius_m=0.04)
```

It should reuse the logic behind `get_max_3d_frm_hm_cube()`:

1. Project multi-view heatmap scores to the renderer's 3D points.
2. Average or robustly aggregate view scores per 3D point.
3. Pick top-K 3D points with 3D NMS.
4. Return:

```text
candidate_xyz:   [B, K, 3]
candidate_score: [B, K]
candidate_valid: [B, K]
```

Why 3D topK instead of per-view 2D topK:

- It avoids duplicate 2D peaks that refer to the same 3D point.
- It matches the current SAM2Act waypoint extraction method.
- It is closer to PerAct/RVT's discretized 3D action detection.

Initial values:

```text
K = 3
nms_radius_m = 0.04 or 0.05
positive_radius_m = 0.05
```

### 5.2 Candidate features

For each candidate `c_i`, build a small feature vector:

```text
f_i = [
  stage1_peak_score_i,
  candidate_xyz_i,
  candidate_xyz_i - prev_selected_xyz,
  gripper_state,
  keyframe/progress embedding if available,
  language embedding projection,
  optional sampled Stage1 feature at candidate projection
]
```

The first implementation should avoid a heavy graph. Use a small MLP or a
one-layer Transformer over K candidates:

```text
candidate_logits = PeakSelector(f_1..f_K, memory_summary)
selected_idx = argmax(candidate_logits)
```

### 5.3 Memory summary

Use the simplest reliable memory first:

```text
memory_summary = MLP([
  previous selected xyz,
  previous action/gripper,
  current proprio,
  pooled language embedding
])
```

Then upgrade to graph memory if needed:

```text
node_t = selected candidate embedding at step t
transition scorer: score(c_i | node_{t-1}, node_t, lang)
```

Local reference code already has useful sketches:

- `stage2_refercodes/idea1_gpsn.py`
  - `PeakExtractor`
  - `PeakNodeEmbedding`
  - `LanguageConditionedTransitionScorer`
- `stage2_refercodes/idea3_dual_memory.py`
  - `extract_topk_peaks`
  - `PeakNodeEmbedding`
  - `peak_selection_loss`
  - `DualLevelMemoryModule`

V9 should reuse the conceptual parts, but implement the minimal path first.

## 6. Training Plan

### 6.1 V9a: oracle-inserted topK selector

Purpose:

```text
Can Stage2/memory learn candidate selection when coverage is guaranteed?
```

Candidate set:

```text
C = {GT candidate} + {hard negatives from Stage1 top peaks or alt targets}
```

Loss:

```text
L_select = CE(selector_logits, gt_candidate_idx)
L_action = normal Stage2 action loss on GT/positive candidate crop
L_total = L_action + lambda_select * L_select
```

Initial value:

```text
lambda_select = 1.0
```

This isolates selector learning from Stage1 coverage failures.

### 6.2 V9b: predicted topK selector

Purpose:

```text
Can the actual Stage1 topK support selection?
```

Candidate set:

```text
C = topK candidates from Stage1 heatmap
```

Label:

```text
positive_i = distance(candidate_i, GT) < 0.05m
```

If multiple positives exist, use a multi-positive target:

```text
L_select = -log sum_i positive_i * softmax(logits)_i
```

If no positive exists:

- log `coverage_miss = 1`
- skip `L_select` for that sample, or insert GT candidate during the scheduled
  transition phase
- keep this as a Stage1 coverage failure, not a Stage2 selector failure

### 6.3 V9c: scheduled oracle-to-predicted topK

Purpose:

```text
Avoid training only on oracle candidates while evaluating on predicted ones.
```

Schedule:

```text
epoch 0-2:   100% oracle-inserted topK
epoch 3-6:    50% oracle-inserted, 50% predicted topK
epoch 7+:     20% oracle-inserted, 80% predicted topK
```

The 20% oracle-inserted floor keeps the selector from forgetting rare/minority
GT candidates.

### 6.4 V9d: K-crop Stage2 reranker

This is the fuller version.

For each candidate:

```text
pc_i = trans_pc(pc, loc=candidate_xyz_i, sca=st_sca)
out_i = mvt2(pc_i)
score_i = selector_head(out_i, candidate_feature_i, memory_summary)
```

Then choose:

```text
selected_idx = argmax(score_i)
action = action_from_mvt2(selected_idx)
```

Loss:

```text
L_select = CE(score_i, positive_idx)
L_action = action loss on positive candidate branch only
L_margin = max(0, margin - score_pos + max(score_neg))
L_total = L_action + lambda_select * L_select + lambda_margin * L_margin
```

Initial values:

```text
K = 3
lambda_select = 1.0
lambda_margin = 0.2
margin = 0.5
```

This is computationally heavier because MVT2 runs K times. It should be tested
after V9a/V9b prove that topK coverage and selection are promising.

## 7. Inference Plan

Initial inference should be conservative:

```text
if topK is unambiguous:
    use top1 as current baseline
else:
    score candidates with selector
    crop around selected candidate
    run mvt2 once
```

Ambiguity detection:

```text
ambiguous if p1/p2 < threshold or candidate_score_entropy > threshold
```

Initial thresholds:

```text
p1/p2 < 20
entropy > 0.5
```

If V9d is enabled:

```text
run mvt2 over all valid K candidates and choose the highest candidate score
```

## 8. Metrics

Report these separately:

| metric | meaning |
|---|---|
| `stage1_top1_gt_hit` | current baseline crop correctness |
| `stage1_topK_gt_hit` | whether V9 has enough candidates |
| `selector_acc_given_coverage` | Stage2 selector quality independent of Stage1 miss |
| `selector_acc_all` | selector quality with coverage misses counted wrong |
| `final_action_error_pos` | local action quality after selected crop |
| `coverage_miss_rate` | true Stage1 bottleneck |
| `top1_fail_topK_hit` | recoverable failures |
| `topK_fail` | unrecoverable by V9 selector |

Important diagnostic:

```text
if top1_fail_topK_hit is high:
    V9 can help.
if topK_fail is high:
    V8/Stage1 proposal quality must improve first.
```

Recommended order:

```text
1. v9-audit on latest Stage1 checkpoint
2. v9a on put_block_back only
3. v9b on put_block_back only
4. v9c on 3 tasks
5. v9d only if v9b shows selector can recover top1 failures
```

## 10. Why This Is Rational

The design is not just intuition:

- BeT supports the "discrete mode + correction" framing.
- C2F Q-attention supports the current coarse-to-fine crop pipeline.
- QTE directly supports expanding top-K coarse candidates under ambiguity.
- LPR supports learning to rank candidate continuations.
- PerAct/RVT support staying in discretized heatmap/3D action-detection space.
- IBC/Diffusion show multimodal imitation is real, but are larger rewrites.

So V9 is the smallest architecture change that directly addresses the observed
failure:

```text
top1 crop is wrong even when another valid/GT candidate may be present.
```

## 11. Decision

Implement V9 in two phases:

```text
V9a/V9b = topK 3D candidate extraction + lightweight selector
V9d     = K-crop Stage2 reranking, only after topK recall is proven useful
```

Do not jump straight to diffusion/EBM or full graph memory. First prove:

```text
Stage1 topK contains recoverable GT candidates.
```

That single number determines whether V9 is the right path.
